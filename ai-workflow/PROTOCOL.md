# 用户可见 Vibe Coding 协议

本协议的目标是让仓库里的 AI 协作过程始终可观察、可解释、可恢复。聊天记录不是唯一事实来源；宿主仓库内 `docs/vibe-coding/` 必须反映当前运行状态。

## 固定文件

| 文件 | 用途 |
|------|------|
| `docs/vibe-coding/README.md` | 用户入口和阅读说明，指向本目录 canonical 文档 |
| `docs/vibe-coding/STATUS.md` | 当前任务状态板 |
| `ZeppOS-Knowledge/ai-workflow/SUBAGENTS.md` | subagent 检索委派规则 |
| `ZeppOS-Knowledge/CONTRIBUTING.md` | 知识消费与回写规则 |
| `docs/vibe-coding/logs/YYYY-MM-DD.md` | 当日阶段日志 |

## 状态更新时机

AI 工具必须在这些节点更新 `STATUS.md`：

1. 接受任务后，开始读取大范围上下文前。
2. 确认任务边界后。
3. 准备编辑文件前。
4. 启动或接收 subagent/explorer 检索任务时。
5. 阶段验证完成后。
6. commit/push 完成后。
7. 发现不能安全提交、需要用户确认或遇到阻塞时。

当阶段有实质进展时，同时追加 `logs/YYYY-MM-DD.md`。

## STATUS.md 字段

`STATUS.md` 必须保持以下结构：

```markdown
# 当前 Vibe Coding 状态

- 状态:
- 当前任务:
- 工作目录:
- 任务边界:
- 正在读取:
- 正在修改:
- 下一步:
- 验证:
- 存档:
- 未纳入本轮的脏改动:
- 更新时间:
```

字段允许简短，但不能删除。

## 透明度规则

- 如果 AI 计划修改文件，先列出预计影响文件。
- 如果 AI 计划用 subagent 检索资料，先写明问题、范围和只读/可写边界。
- subagent 返回后，只把压缩证据包和被采用的结论写入状态或日志，不把大段搜索结果塞进主上下文。
- 如果 AI 计划提交，先说明本轮暂存范围。
- 如果本轮产生可复用 ZeppOS 知识，提交前必须写明回写到 `ZeppOS-Knowledge/self-knowledge/`、项目 `PROJECT.md`，或说明不回写原因。
- 如果工作区已有无关脏改动，明确列入“未纳入本轮的脏改动”。
- 如果验证失败，记录失败命令和下一步修复计划。
- 如果跳过 commit/push，记录用户指令或安全原因。

## 中文优先

面向用户的说明、状态、日志使用中文。代码标识、文件路径、命令、API 名称保持原文。

## 与 Git 自动存档的关系

仓库默认阶段完成后自动 commit/push。状态文件本身也属于工作流变更；如果某轮任务只更新状态日志，通常可以和该轮业务变更一起提交。若用户明确 `no-commit`，仍要更新 `STATUS.md`，但不提交。

为避免无限提交循环，commit/push 之后写入 `STATUS.md` 的最终提交号和推送结果可以作为本地实时状态保留，不强制立刻再提交一次。下一轮有实质变更时再一起存档。若用户要求远端也必须显示最终状态，可单独做一次“状态收尾提交”，但不要再为该收尾提交继续更新提交号。
