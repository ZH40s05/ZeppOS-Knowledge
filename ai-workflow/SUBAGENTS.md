# Subagent 检索委派协议

本协议用于约束文档搜索、案例查找、跨目录扫描等容易膨胀主上下文的任务。目标是让主 agent 保持干净的决策上下文，同时让用户仍然能看懂 AI 工具正在查什么、为什么查、查到了什么。本文件是 canonical 版本；根仓库不再保留 `docs/`。

## 核心原则

- 主 agent 负责目标确认、任务边界、最终判断、文件编辑、验证、commit/push。
- subagent/explorer 负责只读检索、案例枚举、证据整理、独立验证建议。
- subagent 默认不修改文件；需要修改时必须有明确、互不重叠的文件责任边界。
- subagent 的输出必须是压缩后的证据包，不把大段原文、完整搜索结果或无关目录内容带回主上下文。
- 所有面向用户的状态、结论和交接内容使用中文；路径、命令、API 名称保留原文。

## 默认委派的任务

遇到以下情况，AI 工具应优先把检索任务分派给 subagent/explorer：

1. 需要在 `ZeppOS-Knowledge/official/`、`ZeppOS_RE/docs/`、`.claude/memory/` 或多个项目目录中查找资料。
2. 需要查找“有没有类似实现”“哪些项目用过这个 API/模式”“历史上是否记录过这个问题”。
3. 需要比较官方文档、RE 文档、项目文档之间的差异。
4. 需要审计文档是否遗漏、迁移历史 memory、确认知识应写入哪里。
5. 需要独立复核某个规则、skill 或文档变更是否足够明确。

这些任务会消耗大量上下文，但通常可以独立完成，因此不应默认由主 agent 全量展开。

## 不适合委派的任务

以下任务通常保留在主 agent：

1. 任务的下一步立即依赖结果，且检索范围很小。
2. 只需要读取目标文件、邻近配置或一个明确文档。
3. 需要最终权衡产品方向、提交范围或 Git 策略。
4. 涉及账号、密钥、本地隐私文件或不可暴露的外部状态。
5. 用户明确要求不要使用 subagent，或当前工具没有可用的 subagent 能力。

没有 subagent 能力时，仍按本协议的边界本地执行：先 `rg` 精确搜索，再只打开命中文件，并把结果压缩为证据包。

## 委派前可见化

主 agent 在启动 subagent 前应更新 `STATUS.md`：

- `正在读取`: 写明准备委派的资料范围，例如“subagent 检索 ZeppOS 官方 UI 文档和 RE runtime quirks”。
- `下一步`: 写明 subagent 要回答的问题。
- `任务边界`: 写明是否只读、是否允许改文件。

如果并行启动多个 subagent，每个问题必须互相独立，避免重复扫描同一范围。

## 委派提示模板

```text
请在 <仓库路径> 中只读检索，回答这个问题：<具体问题>。

边界：
- 只读取这些范围：<路径/文档类别>。
- 不要修改文件，不要提交。
- 使用 rg 或精确路径定位，不要全量展开大目录。
- 不要读取 NormalApps/其它开发者，除非用户明确要求。

请用中文返回证据包：
1. 结论摘要：3-5 条。
2. 已检查来源：路径列表，必要时带行号或标题。
3. 关键证据：只摘录短句或概括，不粘贴长文。
4. 仍不确定的点：需要测试或用户确认的内容。
5. 建议主 agent 下一步读取的最小文件集合。
```

## 证据包格式

subagent 返回给主 agent 的结果应尽量短，优先包含：

- 结论类型：官方确认、真机/项目经验、RE 确认、推断、仍需测试。
- 来源类别：official、self-knowledge、reverse engineering、project docs、historical memory。
- 最小引用路径：让主 agent 可以按需打开原文件复核。
- 风险或冲突：官方文档与 RE/真机经验不一致时必须标出。

主 agent 不应把 subagent 的长输出原样写进最终回复；应归纳为决策依据，并在需要时打开少量引用文件复核。

## 接收结果后的动作

主 agent 收到结果后：

1. 判断证据是否足够支持下一步。
2. 只读取 subagent 建议的最小文件集合。
3. 将采用的结论写入任务说明、代码注释、项目文档或长期知识文档。
4. 若发现可复用新知识，按知识路由写入 `ZeppOS-Knowledge/self-knowledge/` 或项目 `PROJECT.md`；RE 原始证据链才写入 `ZeppOS_RE/docs/`，不要写入 `.claude/memory/`。
5. 在 `STATUS.md` 或当日日志中记录 subagent 检索完成和采用结论。

## 失败处理

- subagent 超时或无结果时，主 agent 可以关闭该任务并用本地精确搜索继续。
- 如果 subagent 输出过宽、缺少来源或不可复核，主 agent 不应直接采用结论。
- 如果 subagent 发现需要修改文件，主 agent 重新确认写入边界；不要让只读检索任务直接变成无边界编辑任务。
